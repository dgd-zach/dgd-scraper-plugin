Overview

DGD Web Scraper is a WordPress plugin that provides a flexible WP-CLI command to import content from any website into your WordPress site. It automatically:

  • Manages multiple scraper profiles via admin interface
  • Scrapes content from paginated listing pages
  • Extracts titles, dates, content, and images using XPath
  • Removes unwanted elements (ads, scripts, social buttons) from content
  • Creates WordPress posts or pages with proper formatting
  • Imports images to Media Library or sets via FIFU plugin
  • Can add multiple categories and tags
  • Prevents duplicate imports with intelligent skip detection
  • Handles image validation
  • Preserves <pre> tag formatting
  • Schedules automatic imports via built-in WordPress cron
  • Defaults to dry run mode for safety
  • Supports graceful interruption (Ctrl+C)
  • Sets current user as post author
  • Works with saved admin profiles

Installation

Step 1: Install the Plugin

Option A: Via WordPress Admin (Recommended)

  1. Go to Plugins → Add New → Upload Plugin
  2. Click Choose File and select dgd-web-scraper.zip
  3. Click Install Now
  4. Click Activate Plugin

Option B: Via FTP/SSH

  1. Upload the dgd-web-scraper folder to /wp-content/plugins/
  2. Go to Plugins in WordPress admin
  3. Find DGD Web Scraper and click Activate

Step 2: Verify Installation

Check if the plugin is working:

  • Go to Tools → Scraper Settings in WordPress admin
  • You should see the profile management interface

Check if WP-CLI command is available:

wp scraper --help

Step 3: (Optional) Install FIFU Plugin

For external featured images, install the Featured Image from URL (FIFU) plugin:

wp plugin install featured-image-from-url --activate

Or via WordPress admin: Plugins → Add New, search for "Featured Image from URL"

Basic Usage

📁 Where to Run Commands: All WP-CLI commands must be run from your WordPress root directory (the folder containing wp-config.php), not from wp-content or theme folders.

# Navigate to WordPress root first
cd /path/to/wordpress/

# Verify you're in the right place
ls wp-config.php

# Then run scraper commands
wp scraper run --profile=your-profile

Step 1: Configure settings in WordPress admin:

  1. Go to Tools → Scraper Settings
  2. Click ➕ New Profile
  3. Fill in all the XPath selectors and options
  4. Click Save Profile

Step 2: Run the scraper:

# Dry run (preview only)
wp scraper run --profile=your-profile-name

# Live run (actually create posts)
wp scraper run --profile=your-profile-name --live

💡 Dry Run by Default: The scraper runs in dry run mode by default. You must add the --live flag to actually create posts.

Managing Profiles

Creating a Profile

  1. Go to Tools → Scraper Settings
  2. Click ➕ New Profile
  3. Fill in:
    • Profile Name (required) - e.g., "Press Releases"
    • Source URL (required) - Archive page to scrape
    • XPath Selectors (required) - For listing, title, content, date
    • Categories & Tags (optional) - Comma-separated
    • Remove Elements (optional) - One XPath per line
  4. Click Save Profile

Switching Between Profiles

Click on any profile tab at the top of the admin page to switch between profiles.

Editing a Profile

  1. Click the profile tab
  2. Make your changes
  3. Click Save Profile

Duplicating a Profile

  1. Click the profile tab you want to duplicate
  2. Click 📑 Duplicate Profile
  3. A copy will be created with "(Copy)" added to the name

Deleting a Profile

  1. Click the profile tab you want to delete
  2. Click 🗑️ Delete Profile
  3. Confirm the deletion

Listing Profiles via CLI

wp scraper list-profiles

✨ Multiple Profiles: You can create unlimited profiles for different content sources and switch between them easily!

wp scraper run --profile=press-releases --live

Finding XPath Selectors

Using Browser Developer Tools

  1. Open the target website in Chrome or Firefox
  2. Right-click on the element you want (title, content, etc.)
  3. Select "Inspect" or "Inspect Element"
  4. In the Developer Tools panel:
    • Right-click on the highlighted HTML element
    • Select Copy → Copy XPath
  5. Paste this XPath into the admin form

💡 Pro Tip: The admin interface has a built-in XPath guide! Click the 📖 How to find XPath selectors button for detailed instructions and examples.

Testing XPath in Browser Console

Open the browser console (F12) and test your XPath:

// Test an XPath selector:
$x('//article//h1')

// This returns an array of matching elements
// If you see results, your XPath is working!

Common XPath Examples

Element XPath Example
Title in H1 //h1 or //article//h1
Content div //div[@class="content"]
Article body //article
Time/Date //time or //span[@class="date"]
All article links //a[contains(@href, "/articles/")]
Links with class //a[@class="article-link"]

Removing Unwanted Elements

The scraper can automatically remove ads, social buttons, scripts, and other unwanted elements from imported content.

How to Use

  1. In your profile settings, find Remove Elements XPath
  2. Enter XPath selectors, one per line
  3. Save profile

Common Examples

Remove XPath Example
Ads //div[@class="advertisement"]
Social buttons //div[contains(@class, "social-share")]
Scripts //script
Iframes //iframe
Comments //div[@id="comments"]
Related posts //div[contains(@class, "related-posts")]

Example Configuration

In the Remove Elements XPath field:

//div[@class="ads"]
//div[contains(@class, "social-share")]
//script
//iframe
//div[@id="comments"]

✨ Clean Content: These elements will be removed before the content is saved to WordPress, giving you clean, distraction-free posts!

Featured Images

How Featured Images Work

  1. Extracts first image from scraped content (within your content_xpath)
  2. Validates image URL - checks for valid extension (.jpg, .png, etc.)
  3. Attempts to import to Media Library - downloads and saves as WordPress attachment
  4. Falls back to FIFU plugin if import fails (sets external URL)
  5. Uses fallback image if configured and no valid image found

📷 Image Source: Images are extracted from the content area only (not headers/footers) to avoid grabbing logos or navigation images. If you need a specific image outside the content area, use the Featured Image XPath field in your profile.

✅ Valid Image URLs:

https://example.com/images/photo.jpg
https://cdn.example.com/picture.png
https://media.example.com/image.webp

❌ Invalid Image URLs (use fallback if configured):

https://example.com/media?id=12345
https://example.com/document
(any URL without an image file extension)

Setting a Fallback Image

Configure in your profile settings under Fallback Image ID.

To find your image ID:

wp media list --format=table

Finding Posts With Fallback Images

wp post list \
  --meta_key=used_fallback_featured_image \
  --meta_value=true \
  --format=table

Troubleshooting

Problem: "Command not found"

Cause: Plugin not activated or WP-CLI not finding it

Solution:

# Check if plugin is active
wp plugin list | grep generic-web-scraper

# If not active, activate it
wp plugin activate dgd-web-scraper

# Verify command is available
wp scraper --help

Problem: "No content found"

Cause: Listing XPath selector not finding links

Solution:

  1. Test your XPath in browser console: $x('your-xpath')
  2. Make sure you're targeting the right elements
  3. Check if content loads via JavaScript (may need different approach)

Problem: "Failed to extract data"

Cause: Title, content, or date XPath not working

Solution:

  1. Test each XPath selector individually
  2. Inspect the HTML structure of target pages
  3. Try simpler XPath selectors first
  4. Use --debug flag to see detailed output

Problem: "All posts skipped"

Cause: Posts already imported (duplicate detection working)

This is expected behavior. The scraper prevents duplicates by checking the source_url meta field. To re-import, delete existing posts first.

Problem: "Categories/Tags fields not showing in admin"

Cause: Post Type is set to "page"

This is expected behavior. Pages don't support categories and tags by default, so these fields are automatically hidden when you select "page" as the post type.

Problem: "Profile not found"

Cause: Profile name doesn't match saved profiles

Solution:

# List all available profiles
wp scraper list-profiles

# Use the exact profile key shown in the list
wp scraper run --profile=press-releases --live

Automatic Scheduling (WP-Cron)

DGD Web Scraper includes built-in WordPress cron integration that lets you schedule automatic imports directly from the admin interface — no server cron configuration required.

Enabling Automatic Scheduling

  1. Go to Tools → Scraper Settings
  2. Select the profile you want to schedule
  3. Scroll down to the Automatic Scheduling section
  4. Check Enable Automatic Scraping
  5. Set your desired Frequency (1–24) and Period (hour, day, or week)
  6. Click Save Profile

✨ Zero Configuration: Once enabled, WordPress will automatically run the scraper for that profile at the interval you specify. No shell scripts, no system crontab — it just works.

Scheduling Options

Setting Options Description
Enable Automatic Scraping On / Off Toggle scheduled imports for this profile
Frequency 1–24 How many times to run per period
Period Hour / Day / Week The time window for the frequency

Schedule Examples

Frequency Period Result
1 Day Runs once every 24 hours
2 Day Runs every 12 hours
4 Hour Runs every 15 minutes
1 Week Runs once per week
3 Week Runs roughly every 2.3 days

How It Works

  • When you save a profile with scheduling enabled, the plugin registers a WordPress cron event with a custom interval matching your settings.
  • Each scheduled profile gets its own independent cron hook (dgd_scraper_cron_{profile-key}), so multiple profiles can run on different schedules.
  • When the cron event fires, the scraper runs the profile in live mode automatically — importing new content and skipping duplicates just like a manual run.
  • If you change the frequency or period and re-save, the old schedule is cleared and a new one is created automatically.
  • Disabling the checkbox or deleting the profile clears its scheduled events.
  • All scheduled events are cleaned up when the plugin is deactivated.

Viewing the Next Scheduled Run

After saving a profile with scheduling enabled, the admin interface displays the Next Scheduled Run date and time directly below the scheduling options. This shows exactly when WordPress will next execute the scraper for that profile.

💡 WP-Cron Dependency: WordPress cron is triggered by site visits. If your site has very low traffic, scheduled events may not fire precisely on time. For high-precision scheduling on low-traffic sites, set up a system cron to hit wp-cron.php regularly:

*/5 * * * * curl -s https://yoursite.com/wp-cron.php?doing_wp_cron > /dev/null 2>&1

Or use WP-CLI:

*/5 * * * * cd /path/to/wordpress && wp cron event run --due-now > /dev/null 2>&1

Per-Profile Scheduling

Each profile has its own independent schedule. This means you can:

  • Run a "Breaking News" profile every 15 minutes
  • Run a "Weekly Digest" profile once per week
  • Run a "Blog Archive" profile once per day
  • Leave other profiles with scheduling disabled for manual-only use

Debugging Scheduled Events

If you need to verify that schedules are set up correctly, use the included debug script:

wp eval-file wp-content/plugins/dgd-web-scraper/debug-schedule.php

This shows all registered custom intervals, currently scheduled events, and next run times.

Automated Imports with System Cron (Alternative)

💡 Prefer Built-In Scheduling: The plugin now includes built-in WordPress cron scheduling that you can configure directly from the admin interface. The system cron approach below is only needed if you require precise timing or want to run imports outside of WordPress.

Create a bash script (import-content.sh):

#!/bin/bash
# Daily content import

LOG_FILE="/var/log/scraper.log"

echo "Starting import: $(date)" >> $LOG_FILE

# Import from multiple profiles
wp scraper run --profile=press-releases --live >> $LOG_FILE 2>&1
wp scraper run --profile=blog-posts --live >> $LOG_FILE 2>&1

echo "Finished: $(date)" >> $LOG_FILE

Add to crontab to run daily at 2 AM:

0 2 * * * /path/to/import-content.sh

Is Web Scraping Legal?

Yes, with important conditions. In the United States, there is no federal law that prohibits web scraping of publicly accessible data. Recent court rulings have clarified that scraping public information is generally legal, but how you scrape and what you do with the data matters significantly.

✅ Generally Legal:

  • Scraping publicly accessible data (no login required)
  • Using scraped data for personal analysis or research
  • Price monitoring for internal business decisions
  • Market research and competitive analysis
  • Extracting facts and data points (not creative content)

❌ Illegal or High Risk:

  • Scraping content behind login walls or paywalls
  • Republishing copyrighted content (articles, images, creative works)
  • Collecting personal data without legal basis (GDPR/CCPA violations)
  • Bypassing CAPTCHAs or security measures
  • Overloading servers or causing technical harm
  • Violating explicit Terms of Service prohibitions

Key Laws to Be Aware Of

Computer Fraud and Abuse Act (CFAA)

The CFAA prohibits "unauthorized access" to computer systems. Courts have ruled that accessing publicly available data does not violate the CFAA if you:

  • Do not bypass login credentials or access controls
  • Do not circumvent technical barriers (CAPTCHAs, rate limits)
  • Scrape only public, non-protected content

Digital Millennium Copyright Act (DMCA)

The DMCA prohibits circumventing technological measures that protect copyrighted works, such as:

  • Paywalls
  • Login requirements
  • CAPTCHAs
  • Rate limiters

Copyright Law

Copyright protects creative works including articles, blog posts, images, and videos. Key points:

  • Facts are not copyrightable - you can extract factual data
  • Creative expression IS copyrightable - you cannot republish articles
  • Use data for analysis, not republication - transformative use is safer

GDPR (Europe) & CCPA (California)

These privacy laws regulate collection of personal data:

  • Personal Data: Names, emails, phone numbers, addresses, etc.
  • Requirements: Legal basis for collection, transparency, data subject rights
  • Penalties: Up to €20 million or 4% of global revenue (GDPR)

Best Practices for Legal Scraping

1. Check robots.txt

Always check the target site's robots.txt file:

https://example.com/robots.txt

While not legally binding, respecting robots.txt demonstrates good faith and reduces legal risk.

2. Read Terms of Service

Many websites explicitly prohibit scraping in their Terms of Service. Violating ToS can lead to breach of contract claims.

3. Implement Rate Limiting

This tool includes a 0.5-second delay between requests. Keep it enabled to:

  • Avoid overloading servers
  • Prevent "trespass to chattels" claims
  • Reduce risk of IP blocking

4. Only Scrape Public Data

Never scrape:

  • Content behind login walls
  • Password-protected areas
  • Subscription-only content
  • Private user profiles

5. Respect Copyright

DO: Extract facts, prices, statistics, public information

DON'T: Republish full articles, images, creative writing, or copyrighted content

6. Avoid Personal Data

Be extremely cautious when scraping:

  • Email addresses
  • Phone numbers
  • Physical addresses
  • Any personally identifiable information (PII)

Court ruled that scraping publicly available data does not violate the CFAA. This is one of the strongest precedents supporting web scraping in the U.S.

FAQ

Q: How do profiles work?

A: Profiles store all your scraper settings (XPath selectors, categories, etc.) in WordPress. You can create unlimited profiles for different content sources and switch between them easily.

Q: Do I need to specify all XPath selectors in the command?

A: No! Save your settings in a profile, then just run wp scraper run --profile=name --live. Much simpler!

Q: Will this create duplicate posts?

A: No. The scraper checks the source_url meta field before creating posts. If a post with the same source URL exists, it will be skipped.

Q: What happens when I run the scraper multiple times with a batch limit?

A: The scraper intelligently skips already-imported posts and automatically finds new posts to import. You can safely run it multiple times to import large archives incrementally.

Q: Can I assign multiple categories and tags?

A: Yes! Just separate them with commas: News, Press Release, Featured

Q: Who is set as the post author?

A: The WordPress user who runs the WP-CLI command is automatically set as the post author.

Q: What happens to post dates?

A: The scraper extracts the original publication date and uses that as the WordPress post date, preserving chronological order.

Q: Why are header logos not being used as featured images?

A: The scraper only looks for images within the content area (defined by your content_xpath) to avoid grabbing logos, navigation images, or footers. If you need a specific image outside the content area, use the Featured Image XPath field in your profile.

Q: Can I remove ads and unwanted content from imported posts?

A: Yes! Use the Remove Elements XPath field in your profile to specify which elements should be removed (ads, scripts, social buttons, etc.).

Q: Can I scrape password-protected content?

A: No, the scraper only works with publicly accessible pages.

Q: Will this work with JavaScript-heavy sites?

A: The scraper fetches raw HTML. If content is loaded via JavaScript after page load, it won't be captured. You may need a different solution for such sites.

Q: How do I delete imported posts?

A: Use WP-CLI to bulk delete:

wp post delete $(wp post list --meta_key=imported_from --meta_value="scraper" --format=ids)

Q: What happens when I set Posts Per Batch to 0?

A: Setting it to 0 means "process all posts found" within your Start Page + Max Pages range. This is now the default setting.

Q: How does automatic scheduling work?

A: Each profile has its own scheduling settings. Enable the Automatic Scraping checkbox, set a frequency and period, then save. The plugin uses WordPress WP-Cron to run the scraper automatically at the interval you configure. See Automatic Scheduling for full details.

Q: Why isn't my scheduled import running on time?

A: WordPress cron is triggered by site visits. If your site has low traffic, events may fire late. To fix this, set up a system cron to ping wp-cron.php every few minutes. See the WP-Cron Dependency note for instructions.

Q: Can I schedule different profiles at different intervals?

A: Yes! Each profile has independent scheduling settings. You can run one profile hourly and another weekly — they don't affect each other.

Q: What happens to schedules when the plugin is deactivated?

A: All scheduled cron events are automatically cleared when the plugin is deactivated. If you reactivate the plugin, you'll need to re-save profiles to re-register their schedules.

Q: Is this legal?

A: Scraping public content is generally legal in the United States, but important conditions apply. See the Legal & Compliance section for comprehensive information on:

  • What is legal vs. illegal
  • Key laws (CFAA, DMCA, Copyright, GDPR, CCPA)
  • Best practices and compliance checklist
  • When to get legal advice

Key points: Always check robots.txt and Terms of Service, only scrape public data, don't republish copyrighted content, and avoid collecting personal information.
